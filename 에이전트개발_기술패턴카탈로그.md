## 1. 기본 LLM 연동 · 프롬프트 · 구조화 출력
*(출처: Day1)*

| 패턴 | 문제 / 언제 쓰나 | 핵심 API 형태 |
|---|---|---|
| 기본 Chat 호출 | LLM 응답을 받는 가장 기본 단위 | `llm = ChatOpenAI(model=..., temperature=...); llm.invoke("...")` |
| 스트리밍 응답 | 챗봇처럼 토큰이 실시간으로 나와야 할 때 | `for chunk in chain.stream(q): print(chunk.content, end="")` |
| `PromptTemplate` / `ChatPromptTemplate` | 반복되는 질문 틀을 변수화 | `ChatPromptTemplate.from_messages([("system",...), ("user","{q}")])` |
| `MessagesPlaceholder` | 대화 이력을 프롬프트 중간에 가변 길이로 삽입 | `MessagesPlaceholder(variable_name="history")` |
| LangChain Hub 프롬프트 재사용 | 검증된 커뮤니티 프롬프트 활용 (실패 시 로컬 폴백) | `client.pull_prompt("rlm/rag-prompt")` + try/except 폴백 |
| **구조화 출력 (`with_structured_output`)** | LLM 출력이 정해진 스키마를 반드시 지켜야 할 때 (가장 많이 재사용되는 패턴) | `class X(BaseModel): ...`; `llm.with_structured_output(X).invoke(...)` → 검증된 객체 반환 |
| `StrOutputParser` / `JsonOutputParser` / `CommaSeparatedListOutputParser` | 자유 텍스트가 아니라 문자열/JSON/리스트로 바로 받고 싶을 때 | `chain = prompt \| llm \| StrOutputParser()` |
| LCEL 체인 (`\|` 파이프) | prompt→model→parser를 선언적으로 연결 (레거시 `LLMChain` 대체) | `chain = prompt \| llm \| parser` |
| 순차 체인 (다단계) | 한 체인의 출력을 다음 체인 입력으로 (레거시 `SimpleSequentialChain` 대체) | `{"summary": chain1} \| prompt2 \| llm \| parser` |
| `RunnablePassthrough` | 원본 입력값을 그대로 다음 단계로 흘려보내야 할 때 (RAG의 question 분기 등) | `{"context": retriever\|format_docs, "question": RunnablePassthrough()} \| prompt \| llm` |
| `RunnableWithMessageHistory` | 세션별 대화 메모리를 체인에 자동 연결 (레거시 `ConversationChain` 대체) | `RunnableWithMessageHistory(chain, get_session_history, ...)` + `configurable.session_id` |
| API 키 관리 | 평문 하드코딩 대신 안전하게 키 로드 | `getpass()` 입력 또는 `.env` + `load_dotenv()` — **하드코딩 폴백 금지** |

## 2. 문서 처리 · RAG 구축
*(출처: Day1, Day2)*

| 패턴 | 문제 / 언제 쓰나 | 핵심 API 형태 |
|---|---|---|
| `PyPDFLoader` / `PyMuPDFLoader` | PDF를 페이지 단위 `Document`로 변환, 페이지 출처를 답변에 표기 | `loader.alazy_load()`; `Document(metadata={"page_number":...})` |
| `WebBaseLoader` + `bs4.SoupStrainer` | 웹페이지에서 본문만 스코프해서 노이즈(광고/네비) 제거 | `WebBaseLoader(bs_kwargs=dict(parse_only=SoupStrainer(class_="article")))` |
| 문서유형별 분할 전략 | 균일 `chunk_size` 분할이 약관 조항을 잘라먹는 문제 해결 | 약관: 정규식 전방탐색(`re.split(r"(?=제\d+조)")`), FAQ: 빈 줄 분리, 표: 행→자연어 문장 변환 |
| `RecursiveCharacterTextSplitter` | 일반 문서 청킹, chunk_size/overlap 트레이드오프 이해 필요 | `chunk_size=500~1000`, `chunk_overlap`으로 경계 정보 손실 방지 |
| 적재 전 PII 마스킹 | Vector DB 자체에 민감정보가 그대로 들어가는 것 방지 | 정규식 `PII_PATTERNS` → `re.sub`로 마스킹 후 임베딩 |
| Metadata 설계 + 필터 검색 | 문서유형/상품명/버전별로 좁혀서 검색 | `similarity_search(..., filter={"$and":[{"doc_type":"약관"},{"상품명":...}]})` |
| `OpenAIEmbeddings` 모델 선택 | 속도/비용 vs 정확도 트레이드오프 | 일반 RAG: `text-embedding-3-small`, 고정밀(법률/금융): `text-embedding-3-large` |
| `Chroma.from_documents` + `as_retriever` | 임베딩 저장 + top-k 검색을 표준 Runnable로 노출 | `vectorstore.as_retriever(search_kwargs={"k":4})` |
| `format_docs` 헬퍼 | 검색된 Document 리스트를 프롬프트 `{context}`용 문자열로 변환 | `"\n".join(d.page_content for d in docs)` |
| 자가 검증 체인 (Self-Verification) | 1차 답변을 근거와 나란히 재검토·수정 | LCEL 딕셔너리로 `초안생성`과 `근거검색`을 병렬 실행 → 검증 프롬프트로 재작성 |
| 장문 요약 3대 전략 | 문서가 컨텍스트 창보다 클 때 요약 방식 선택 | Stuff(한번에), Refine(순차 보완), Map-Reduce(청크별 병렬요약 `batch()` 후 통합) |
| 유사도 임계값 (환각 방지) | 근거 없는 질문에 억지로 답하는 것 방지 — **금융권 핵심 안전장치** | `search_type="similarity_score_threshold"` + `score_threshold` 미달 시 "근거 없음" 응답 |

## 3. 검색 품질 개선 · 하이브리드/그래프 검색
*(출처: Day2)*

| 패턴 | 문제 / 언제 쓰나 | 핵심 API 형태 |
|---|---|---|
| `BM25Retriever` | 상품코드·조항번호 등 정확 매칭이 중요할 때 (임베딩 없이 단어빈도 기반) | `BM25Retriever.from_documents(docs)` |
| `EnsembleRetriever` (하이브리드 검색) | BM25 + 벡터 검색을 가중 결합 — 금융문서 RAG의 실무 표준 구성 | `EnsembleRetriever(retrievers=[bm25, vector], weights=[0.5,0.5])` |
| GraphRAG (지식그래프 + Vector 융합) | Vector 단독 검색이 "상품→예외조항" 같은 연결관계를 놓칠 때 | `networkx.DiGraph`로 관계 그래프 구축 → Vector로 출발점 찾고 BFS(N-hop)로 확장 |
| LLM 트리플 자동 추출 | 그래프를 수작업이 아니라 문서에서 자동 구축 | `with_structured_output(TripleList)`로 (head, relation, tail) 추출 |
| chunk_size/top-k/필터/임계값 실험 | "감으로" 튜닝하지 않고 값별 비교표로 결정 | 여러 조합을 `batch()` 또는 반복문으로 돌려 결과 비교 |

## 4. Tool / Function Calling · MCP
*(출처: Day2, Day3)*

| 패턴 | 문제 / 언제 쓰나 | 핵심 API 형태 |
|---|---|---|
| Chat Completions Tool Calling 4단계 | LLM이 외부 함수를 호출해야 할 때의 표준 흐름 | ①`tools=[...]` JSON Schema 정의 → ②`tool_choice="auto"` 호출 → ③`tool_calls` 확인 후 함수 실행 → ④결과를 `role:"tool"` 메시지로 추가 후 재요청 |
| `@tool` 데코레이터 (LangChain) | Python 함수를 Agent가 쓸 수 있는 도구로 등록 | `@tool\ndef 함수(query: str) -> str: """설명(도구 선택 정확도를 좌우함)"""` |
| 함수매핑 딕셔너리 | 여러 도구 중 자동 선택 시 if/elif 분기 대신 확장 가능한 구조 | `available_functions = {"이름": 함수}` |
| Responses API (최신 대안) | Chat Completions보다 평평한 구조, 최종 텍스트 편의 속성 | `client.responses.create(input=..., tools=...)` → `response.output_text` |
| `ToolNode` + `tools_condition` (LangGraph) | "도구가 필요하면 도구로, 아니면 바로 답변" 조건부 분기 | `add_conditional_edges('chatbot', tools_condition)` — `tools_condition`이 자동으로 tools/END 분기 |
| Text-to-SQL | 자연어 질문을 SQL로 변환해 DB 조회 | `create_sql_query_chain(llm, db)` + SQL 클리닝 후처리(````sql` 코드블록 제거) 필수 |
| Pandas DataFrame Agent | CSV/표 데이터에 자연어로 질의·시각화 | `create_pandas_dataframe_agent(llm, df, allow_dangerous_code=True)` — 보안 주의 필요 |
| MCP 서버 작성 (FastMCP) | 도구를 앱 내부가 아니라 독립 서버로 노출 (재사용 가능한 개방 표준) | `mcp = FastMCP("이름"); @mcp.tool()\ndef ...` |
| MCP 클라이언트 (stdio / streamable-http) | 로컬 프로세스 도구 vs 상시운영 원격 서버 | `StdioServerParameters` 또는 `streamablehttp_client(url)` — 도구 호출 코드는 동일, transport만 교체 |
| LangChain Agent + MCP 통합 | MCP 도구를 LangChain Agent에 바로 연결 | `MultiServerMCPClient({...}).get_tools()` → `create_agent(model, tools)` |
| 이종 도구 통합 (DB+RAG+API를 한 Agent에) | 도메인이 바뀌어도 구조는 동일, 도구만 교체 | `@tool`로 통일된 인터페이스에 DB조회/RAG검색/Mock API를 모두 등록 |

## 5. LangGraph 흐름 제어 패턴
*(출처: Day3 — 이번 실습에서 직접 다룬 핵심 영역)*

| 패턴 | 문제 / 언제 쓰나 | 핵심 API 형태 |
|---|---|---|
| **Router (조건부 분기)** | 입력을 분류해서 서로 다른 처리 노드로 보내야 할 때 (콜센터 ARS 비유) | `route_decision(state) -> Literal[...]`; `add_conditional_edges('route', route_decision, {...})` |
| **Map-Reduce (`Send`)** | 입력이 여러 독립 하위작업으로 쪼개지고 각각 LLM 호출이 필요할 때 (보고서 챕터별 작성 등) | `def assign_workers(state): return [Send("llm_call", {"chapter": c}) for c in state["contents"]]` |
| **Evaluator-Optimizer (생성-평가 반복)** | 생성물을 스스로 평가해서 기준 미달이면 피드백 반영 후 재생성 | `evaluator = llm.with_structured_output(Feedback)`; `route_code` 함수로 `Accepted`/`Rejected+Feedback` 분기 → 재생성 루프 |
| **검증 루프 (3중 검문소)** | 답변이 나가기 전 규칙기반+LLM심사를 순차 통과시켜야 할 때 | 입력 유효성(규칙) → 근거 검증(LLM 심사) → 출력 검사(규칙, 금지표현/PII) → `retry` 카운터로 무한루프 방지 |
| **Human-in-the-loop (`interrupt` + `Command`)** | 그래프 실행 중 사람의 승인/입력을 기다려야 할 때 (송금 승인 등 — "마지막 버튼은 사람이 누른다") | `interrupt({...})`로 멈춤 → `"__interrupt__"` 감지 → `Command(resume=값)`으로 재개, `Command(goto="노드")`로 흐름 이동 |
| Checkpointer (`MemorySaver`/`InMemorySaver`) | 멈춘 지점의 상태를 기억해야 재개가 가능 | `builder.compile(checkpointer=MemorySaver())` + `thread_id`로 세션 구분 |
| `Annotated[T, operator.add]` (Reducer) | 노드가 반환한 값을 덮어쓰지 않고 누적해야 할 때 (재시도 횟수 카운트 등) | `retry_count: Annotated[int, operator.add]` — `{"retry_count": 1}` 반환 시 자동 누적 |
| 의도분류 기반 RAG 라우팅 | 질문 유형(정책/직원/양식 등)에 따라 다른 벡터DB를 검색해야 할 때 | `IntentResult(BaseModel)`로 `target_db: Literal[...]` 구조화 분류 → 해당 retriever로 라우팅 |
| Pydantic vs TypedDict 구분 | LLM 출력 검증(런타임 오류 필요) vs 그래프 상태 전달(가벼운 컨테이너) | 출력 검증엔 `BaseModel`, 그래프 State엔 `TypedDict` |

## 6. 멀티에이전트 오케스트레이션
*(출처: Day4)*

| 패턴 | 문제 / 언제 쓰나 | 핵심 API 형태 |
|---|---|---|
| 고정 경로(Ping-pong) 협업 | 흐름 순서가 절대 바뀌지 않는 단순 2단계 작업 (예: 조사→시각화) | 각 노드가 `Command(goto=다음노드)` 반환, 완료 신호 문자열로 `END` 분기 |
| **Supervisor 패턴 (구조화 출력 라우팅)** | 다음 작업자가 동적으로 결정돼야 할 때 (조건부 스킵/재시도/작업자 추가가 쉬움) | `Router(BaseModel)`로 `dest: Literal[*routes]` 강제 → `Command(goto=dest, update={...})` |
| 검증 단계를 필수 경유시키는 Supervisor | 모든 답변이 나가기 전 반드시 검증(verify) 워커를 거쳐야 할 때 | 시스템 프롬프트에 "작업자 답변 후 반드시 verify로" 명시 + `next_worker` 상태 필드 |
| `recursion_limit` + `GraphRecursionError` 처리 | Supervisor가 FINISH를 못 찾고 무한 루프에 빠질 위험 방지 | `graph.stream(..., {"recursion_limit": 15})` + `try/except GraphRecursionError` |
| 메시지 태깅 (누가 말했는지 기록) | 여러 에이전트가 하나의 messages 리스트를 공유할 때 출처 추적 | `HumanMessage(content=ans, name="product")` |
| `Command(goto=..., update=...)` 통합 전이 | 조건부 엣지 함수 없이 노드 스스로 "무엇이 바뀌었고 어디로 갈지"를 함께 선언 | 그래프 배선이 `START → supervisor` 하나로 단순화됨 |
| Mock 데이터 기반 도메인 도구 | 실제 DB/검색엔진 연동 전에 프로토타입 검증 | 딕셔너리 키워드 매칭으로 실제 서비스 도구를 흉내 |
| 역할별 시스템 프롬프트 팩토리 | 여러 에이전트가 공통 프로토콜(핸드오프 신호 등)은 공유하고 역할만 다를 때 | `make_system_prompt(suffix)` 함수로 공통 뼈대 + 역할별 문구만 주입 |
| RAG+Tool+Memory 통합 Agent | 서비스화 이전, 여러 능력(검색/도구/메모리)을 한 Agent로 조립 | `create_react_agent(llm, tools=[...], checkpointer=MemorySaver())` |

## 8. 운영 복원력 (Fallback · Rollback · Retry)
*(출처: Day5)*

| 패턴 | 문제 / 언제 쓰나 | 핵심 API 형태 |
|---|---|---|
| 모델 폴백 체인 | 주 모델 호출 실패 시 서비스가 눈에 띄게 죽으면 안 될 때 | `primary.with_fallbacks([secondary, final_backup])` |
| Timeout + Retry | 무한 대기 방지(timeout) + 일시적 오류 자동 복구(retry) | `ChatOpenAI(request_timeout=10, max_retries=0).with_retry(stop_after_attempt=2)` |
| 수동 폴백 + 로깅 | 어떤 모델이 실제로 응답했는지 추적(장애 진단/SLA용) | try/except로 Primary/Secondary 여부를 명시적으로 로그 |
| 폴백 트리거 기준표 (계획 문서용) | "언제 폴백을 발동할지"에 대한 설계 기준 | Timeout / 오류율 / 비용 / 점검시간 각각에 대응 조치 매핑 |
| **상태머신 기반 Fallback+Rollback** | 다단계 파이프라인(검색→초안→검증→답변)에서 실패 지점별로 다르게 대응 | 검색 실패→키워드 폴백→사람 이관, 검증 실패→`rollback`으로 초안 폐기 후 재검색, `MAX_ROLLBACK`으로 무한루프 방지 |
| Fallback vs Rollback 구분 | 개념 혼동 방지 (계획 문서에 정의해두면 좋음) | Fallback = 수단 실패 → 대안 수단, Rollback = 산출물이 이미 틀림 → 폐기 후 이전 단계로 |
