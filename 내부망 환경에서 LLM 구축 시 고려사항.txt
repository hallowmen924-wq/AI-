# 내부망 환경에서 LLM 구축 시 고려사항

내부망에서 LLM을 구축할 때는 모델 성능뿐 아니라 **패키지 설치, 모델 반입, 데이터 보안, 운영 환경**을 함께 고려해야 합니다.

## 1. 내부망 환경을 먼저 확인합니다

개발 전에 다음 항목을 확인해야 합니다.

* 인터넷 접속 가능 여부
* 운영체제와 Python 버전
* GPU와 CUDA 환경
* 외부 API 사용 가능 여부
* 파일과 패키지 반입 절차

외부 환경에서 잘 실행되더라도 내부망 환경이 다르면 코드가 작동하지 않을 수 있습니다.

---

## 2. LLM 운영 방식을 결정합니다

내부망에서는 일반적으로 다음 세 가지 방식이 사용됩니다.

### 외부 LLM API 사용

OpenAI, Claude, Azure OpenAI 등 승인된 외부 API를 사용하는 방식입니다.

구축은 간단하지만 입력 데이터가 외부로 전송될 수 있으므로 보안 검토가 필요합니다.

### 내부망에 오픈소스 LLM 설치

오픈소스 모델을 내부 GPU 서버에 설치하는 방식입니다.

데이터를 외부로 보내지 않을 수 있지만 GPU 자원과 운영 인력이 필요합니다.

### 사내 LLM 플랫폼 사용

기관에서 이미 구축한 사내 생성형 AI 플랫폼이나 LLM API를 사용하는 방식입니다.


---

## 3. 패키지는 미리 준비합니다

내부망에서는 인터넷을 이용해 바로 패키지를 설치하기 어렵습니다.

따라서 외부망에서 필요한 패키지와 의존성을 미리 다운로드해야 합니다.

```bash
pip download -r requirements.txt -d wheelhouse
```

내부망에서는 다운로드한 패키지를 이용해 설치합니다.

```bash
pip install --no-index \
  --find-links=./wheelhouse \
  -r requirements.txt
```

패키지 버전은 반드시 고정하는 것이 좋습니다.

```text
pandas==2.2.3
fastapi==0.115.0
langchain-core==0.3.15
```

---

## 4. 모델 파일도 사전에 반입합니다

내부망에서는 Hugging Face 모델을 자동으로 다운로드하지 못할 수 있습니다.

따라서 모델, 토크나이저, 설정 파일을 외부에서 미리 준비해야 합니다.

```python
model = SentenceTransformer(
    "./models/bge-m3",
    local_files_only=True
)
```

모델을 반입할 때는 라이선스, 출처, 버전, 파일 크기도 함께 확인해야 합니다.

---

## 5. (옵션)Docker 환경도 미리 준비합니다

Docker는 Python, 패키지, 실행환경을 하나로 묶어 동일한 환경에서 실행하도록 해주는 도구입니다.

내부망에서는 Docker 이미지를 인터넷에서 바로 내려받기 어려우므로 외부망에서 미리 저장해야 합니다.

```bash
docker pull python:3.11-slim
docker save python:3.11-slim -o python-image.tar
```

내부망에서는 저장된 이미지를 불러옵니다.

```bash
docker load -i python-image.tar
```

Docker를 사용하지 않는 프로젝트라면 Python 가상환경과 패키지 파일만 준비해도 됩니다.

---

## 6. 데이터 보안과 권한을 설계합니다

내부망에 있는 데이터라고 해서 모든 사용자가 접근할 수 있는 것은 아닙니다.

다음 사항을 확인해야 합니다.

* 개인정보 포함 여부
* 부서별 문서 접근 권한
* 질문과 답변 로그 저장 여부
* 고객정보 마스킹 여부
* 외부 LLM으로 전송 가능한 데이터 범위

RAG 시스템에서는 사용자가 접근할 수 있는 문서만 검색해야 합니다.

```text
사용자 인증
→ 문서 접근 권한 확인
→ 허용된 문서 검색
→ LLM 답변 생성
```

---

## 7. 작은 PoC부터 시작합니다

처음부터 복잡한 Agent 시스템을 구축하기보다 작은 기능부터 검증하는 것이 좋습니다.

```text
1단계: 문서 업로드
2단계: 문서 분할
3단계: 임베딩과 검색
4단계: LLM 답변 생성
5단계: 출처 표시
6단계: 사용자 권한 적용
```

각 단계가 정상적으로 작동하는지 확인한 후 기능을 확장해야 합니다.

---

## 8. RAG는 검색 품질이 중요합니다

RAG에서 잘못된 답변이 나오는 원인은 모델보다 검색 과정에 있는 경우가 많습니다.

다음 항목을 확인해야 합니다.

* 문서가 정상적으로 파싱되었는가
* 정답 문서가 검색되었는가
* 최신 문서가 우선 검색되는가
* 답변과 근거 문서가 일치하는가
* 출처가 정확하게 표시되는가
* 근거가 없을 때 답변을 거절하는가

---

## 9. Agent는 실행 권한을 제한합니다

Agent가 고객정보 조회, 이메일 발송, 데이터 수정 등의 업무를 직접 수행하면 위험할 수 있습니다.

```text
LLM이 작업 제안
→ 입력값 검증
→ 사용자 승인
→ 허용된 도구 실행
→ 실행 기록 저장
```

중요한 업무는 반드시 사람이 최종 확인하도록 설계해야 합니다.

---

## 10. 장애 상황도 준비합니다

내부망에서는 패키지, 모델, GPU, API 문제로 서비스가 중단될 수 있습니다.

다음과 같은 대체 방안을 준비하는 것이 좋습니다.

```text
GPU 장애 → 소형 모델 사용
LLM 장애 → 검색 결과만 제공
패키지 설치 실패 → 사전 구성 환경 사용
모델 반입 지연 → 기존 승인 모델 사용
```

---

# 핵심 정리

내부망 LLM 구축에서 가장 중요한 것은 다음 다섯 가지입니다.

1. 내부망의 운영 환경을 먼저 확인합니다.
2. 패키지와 모델을 미리 준비합니다.
3. 데이터 권한과 보안을 먼저 설계합니다.
4. 작은 PoC부터 단계적으로 구축합니다.
5. 가장 큰 모델보다 안정적으로 운영 가능한 모델을 선택합니다.

> 내부망 LLM 구축의 핵심은 모델만 설치하는 것이 아니라, 패키지·모델·데이터·보안·운영 환경이 함께 작동하도록 만드는 것입니다.
