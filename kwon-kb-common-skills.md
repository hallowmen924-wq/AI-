📋 먼저 아래 4단계 전체 구조를 사용자에게 그대로 보여준 뒤 STEP 0부터 시작해줘.
각 단계를 진행할 때는 응답 맨 앞에 반드시 해당 단계 제목(예: "## STEP 0. 탐색")을 그대로 표시하고 시작해줘.

⚖️ 아래 원칙들은 신중함을 속도보다 우선시해. 다만 오타 수정 같은 사소한 작업에는 판단해서 과하게 적용하지 마.

## 회사 개발 기준
- 공통 모듈을 우선적으로 사용해라. LLM 호출, 설정 로딩, Retrieval, Ingestion, Observability 같은 기능은 직접 재구현하지 말고 공통 모듈의 공개 인터페이스를 사용해라.
- 회사 환경에서는 사내 패키지/사내 PyPI/내부 저장소 기준으로 설치·실행 흐름을 따른다. 외부 라이브러리 의존성은 팀 기준에 맞는 방식으로만 사용해라.
- 설정은 코드에 하드코딩하지 말고 YAML, 환경변수, 시크릿/설정 파일로 관리해라.
- 민감정보(토큰, DSN, 비밀값)는 저장소에 넣지 말고 환경변수 또는 회사 정책에 맞는 시크릿 경로를 사용해라.
- 로컬 개발 시에도 공통 모듈의 예제와 문서를 먼저 참고해라. 예제/문서가 있으면 그 흐름을 우선 재사용해라.
- 구현은 “공통 모듈을 잘 쓰는 방식”이 우선이고, 그 위에 회사 비즈니스 로직만 얹는 형태로 진행해라.
- 바이브코딩으로 빠르게 구현하더라도, 공통 모듈을 우회하는 직접 구현은 피하고 이유를 명확히 설명할 수 있어야 한다.
- 다른 사람이 코드를 읽고 유지보수할 수 있도록, 각 함수/클래스/설정값은 역할이 분명해야 한다.
- 코드 리뷰를 쉽게 하기 위해, 공통 모듈 사용 이유와 비즈니스 로직 분리 포인트를 주석이나 구조로 드러내라.
- “한 번에 통짜로 구현”보다, 작은 단위로 나누고 각 단계마다 왜 공통 모듈을 쓰는지 설명하는 방식으로 진행해라.

## STEP 0. 탐색
관련 코드, 문서, 예제, 공통 모듈 API를 읽고 현재 구조와 회사 개발 기준을 이해해줘. 아직 아무것도 만들지 마.

다음 항목을 우선 확인해줘.
- 공통 모듈의 README, 설치 문서, 설정 문서, 예제 코드
- 현재 작업과 직접 관련된 공통 모듈의 입력/출력 인터페이스
- 회사 환경에서 필요한 설정 방식(예: YAML, 환경변수, 내부 endpoint, DB DSN, 인증 정보)
- 기존 예제가 있다면 그 실행 방식과 설정 파일 규칙

**[가정 명시화]** 요청이 모호하거나 여러 해석이 가능하면, 임의로 하나를 골라 진행하지 말고 그 해석들을 제시하거나 질문해줘. 확신 없는 부분은 추측하지 말고 물어봐.

## STEP 1. 계획
이해한 내용을 바탕으로 무엇을, 어떤 순서로 만들지 계획을 세워줘.
- 입력/출력이 무엇인지
- 공통 모듈에서 어떤 기능을 재사용할지
- 회사 환경에서 어떤 설정/시크릿/엔드포인트가 필요한지
- 각 단계별로 "무엇을 확인하면 성공인지" 검증 기준을 함께 적어줘 (예: "1. [작업] → 확인: [검증 방법]")

계획만 보여주고 코드는 아직 만들지 마.

**[단순함 우선]** 요청받지 않은 기능/추상화/설정 옵션을 넣지 마. 1회성 코드에는 확장성을 고려하지 마. 일어나지 않을 상황에 대한 예외 처리를 넣지 마. "시니어 개발자가 보면 과하다고 할까?"를 기준으로 판단해줘.

## STEP 2. 구현
합의된 계획대로 구현해줘.

다음 원칙을 지켜줘.
- 공통 모듈의 공개 인터페이스를 우선 사용해라. 직접 구현한 LLM 호출, 설정 로더, Retrieval 코드가 있으면 공통 모듈 방식으로 바꿔라.
- 회사 개발 컨벤션이 있다면 그에 맞춰 스타일을 맞춰라. 기존 코드 스타일을 따르고, 불필요한 리팩터링은 하지 마라.
- 설정/시크릿/엔드포인트는 회사 규칙에 맞춰 분리해라.
- 변경된 모든 줄은 사용자의 요청과 직접 연결되어야 해.
- 공통 모듈을 우회하는 구현을 발견하면, 그 이유를 설명하고 가능한 경우 공통 모듈 기반 방식으로 수정해라.
- 리뷰어가 빠르게 이해할 수 있도록, 중복 로직은 줄이고 역할별로 함수를 나눠라.
- 구현 중간에 임시 코드나 하드코딩이 생기면, 반드시 TODO/설명과 함께 남기고 이후 정리할 수 있게 해라.

**[외과적 수정]** 요청과 무관한 코드는 건드리지 마 — 스타일이 마음에 안 들어도 기존 스타일을 따르고, 깨지지 않은 코드를 리팩터링하지 마. 네 변경으로 안 쓰게 된 import/변수만 정리하고, 원래 있던 죽은 코드는 임의로 지우지 말고 언급만 해줘.

## STEP 3. 검증
직접 실행해서 결과가 계획한 대로 맞는지 확인해줘.

다음 항목을 확인해줘.
- 공통 모듈 기반으로 정상 동작하는지
- 설정 파일/환경변수/시크릿이 올바르게 반영되는지
- 예제 실행 또는 테스트가 가능한지
- 회사 개발 환경 기준으로 필요한 네트워크/인증/DB 연결이 가능한지
- 공통 모듈을 우회한 구현이 남아 있지 않는지
- 코드가 다른 사람의 리뷰를 받기 쉬운 구조인지

(사람 검증이 필요한 부분이 있으면 무엇을 확인하면 되는지 구체적으로 알려줘.)

**[목표-검증 루프]** STEP 1에서 정한 검증 기준이 실제로 충족될 때까지 확인·수정을 반복해줘. "일단 돌아간다"가 아니라 계획한 성공 기준을 실제로 만족하는지가 기준이야.
