# 개념: 검색 품질 개선

## 한눈에 보기

RAG 답변이 이상할 때 원인은 대부분 LLM이 아니라 **검색**입니다. "잘 찾아야 잘 답한다."
이 문서는 검색 품질을 좌우하는 5가지 손잡이 — chunk 크기, top-k, 메타데이터 필터, 유사도 임계값(근거 없음 처리), BM25 하이브리드 — 를 실험 코드와 함께 정리합니다.

## 핵심 개념

| 손잡이 | 조절하는 것 | 잘못 설정하면 |
|---|---|---|
| **chunk_size / overlap** | 조각의 크기와 겹침 | 너무 크면 노이즈 혼입, 작으면 문맥 단절 |
| **top-k** | 가져올 조각 개수 | 작으면 정답 누락, 크면 토큰 낭비·주의 분산 |
| **metadata filter** | 검색 대상 범위 | 없으면 다른 상품 문서가 섞여 들어옴 |
| **score threshold** | "근거 없음" 판정선 | 없으면 관련 없는 문서로 환각 생성 |
| **BM25 + 벡터** | 키워드 매칭 + 의미 매칭 | 벡터만 쓰면 상품코드·조항번호를 못 찾음 |

**벡터 검색 vs BM25 — 서로의 빈틈**

| | 강한 질문 | 약한 질문 |
|---|---|---|
| **벡터(의미)** | "노후 대비로 목돈을 굴리고 싶어요" → "은퇴 후 생활자금" 문서를 찾음 | "KB-DEP-2024-017" 같은 코드는 의미가 없어 헤맴 |
| **BM25(키워드)** | 상품코드·조항번호·고유명사를 정확히 매칭 | 문서에 "노후"라는 단어가 없으면 못 찾음 |

→ 그래서 **둘 다 씁니다(Hybrid Search)**.

## 전체 동작 흐름

```
[질문]
  ├─ BM25Retriever    (단어 빈도 기반, 임베딩 불필요)
  └─ Vector Retriever (Chroma, 의미 기반)
        ↓ EnsembleRetriever(weights=[0.5, 0.5]) 로 결과 융합
  [후보 조각들]
        ↓ metadata filter 적용
        ↓ 유사도 임계값 검사 ── 최고 점수 < THRESHOLD ─→ "근거를 찾지 못했습니다" (환각 차단)
  [근거 조각]
        ↓ 프롬프트 결합 → LLM → 답변
```

## 핵심 코드

### 1. chunk 파라미터 실험

값을 바꿔 가며 조각 수와 검색 결과를 눈으로 비교하는 것이 요령입니다.

```python
def split_docs(chunk_size, chunk_overlap):
    splitter = RecursiveCharacterTextSplitter(chunk_size=chunk_size, chunk_overlap=chunk_overlap)
    return splitter.split_documents(documents)

for size, overlap in [(80, 0), (80, 30), (200, 40)]:
    print(f"chunk_size={size:>3}, overlap={overlap:>2}  ->  조각 {len(split_docs(size, overlap))}개")
```

### 2. 거리 지표 명시와 점수 확인

Chroma는 기본 거리 지표가 컬렉션마다 다를 수 있으므로 `hnsw:space`를 명시하는 것이 안전합니다.

```python
store = Chroma.from_documents(
    chunks, embeddings,
    collection_name="kb_base",
    collection_metadata={"hnsw:space": "cosine"},   # 코사인 거리로 고정
)

for doc, score in store.similarity_search_with_score("정기예금을 만기 전에 깨면?", k=3):
    print(f"[점수 {score:.3f}] {doc.metadata['상품명']} | {doc.page_content[:35]}...")
```

> ⚠️ `similarity_search_with_score`는 **거리**(작을수록 유사), `similarity_search_with_relevance_scores`는 **관련도**(클수록 유사)를 돌려줍니다. 임계값 비교 방향을 헷갈리면 로직이 정반대로 동작합니다.

### 3. 메타데이터 필터 — 단일 조건과 복합 조건

```python
# 단일 조건
store.similarity_search_with_score(q, k=2, filter={"상품명": "KB주택담보대출"})

# 복합 조건 — Chroma는 $and / $or 연산자를 씁니다
store.similarity_search_with_score(
    q, k=2,
    filter={"$and": [{"상품명": "KB주택담보대출"}, {"문서종류": "약관"}]},
)
```

### 4. "근거 없음" 처리 — 금융권 RAG의 핵심 안전장치 ⭐

문서에 근거가 없는 질문인데 LLM이 그럴듯하게 지어내면 큰 사고입니다. 검색 단계에서 먼저 막습니다.

```python
THRESHOLD = 0.4

def retrieve_with_guard(question, k=3):
    hits = store.similarity_search_with_relevance_scores(question, k=k)
    best = hits[0][1]
    if best < THRESHOLD:
        return None, best          # 근거 부족 → 답변 생성 자체를 하지 않음
    return hits, best

def answer(question):
    hits, best = retrieve_with_guard(question)
    if hits is None:
        return f"답변할 근거를 찾지 못했습니다. (최고 유사도 {best:.3f} < 임계값 {THRESHOLD})"
    context = "\n".join(d.page_content for d, _ in hits)
    return answer_chain.invoke({"context": context, "question": question})
```

Retriever 레벨에서 처리하는 방법도 있습니다. 체인에 그대로 꽂을 수 있어 더 간결합니다.

```python
guard_retriever = store.as_retriever(
    search_type="similarity_score_threshold",
    search_kwargs={"score_threshold": THRESHOLD, "k": 3},
)
guard_retriever.invoke("KB 자동차보험 자기부담금은?")   # → 조각 0개 반환
```

이 "정직한 실패" 사상은 Day3의 루프 탈출, Day5의 폴백 설계로 계속 이어집니다. → [[개념_Agent_통제력과_루프탈출]]

```python
try:

except ImportError:

vector_retriever = vector_db.as_retriever(search_kwargs={"k": 2})
bm25_retriever = BM25Retriever.from_documents(hybrid_docs)
bm25_retriever.k = 2

hybrid_retriever = EnsembleRetriever(
    retrievers=[bm25_retriever, vector_retriever],
    weights=[0.5, 0.5],       # 가중치로 어느 쪽을 더 믿을지 조절

```

### 6. 한국어 형태소 토크나이저 연결 (Kiwi)

BM25는 공백 기준 토큰화가 기본이라 한국어 조사·어미 때문에 매칭률이 떨어집니다. Day3 실습에서는 `kiwipiepy`를 붙입니다.

```python
from kiwipiepy import Kiwi
kiwi = Kiwi()

def kiwi_tokenize(text):
    return [token.form for token in kiwi.tokenize(text)]

bm25 = BM25Retriever.from_documents(split_docs, preprocess_func=kiwi_tokenize)
```

가중치는 데이터 성격에 맞춥니다. Day3 실습 기준 — 규정집 `[0.5, 0.5]`, **직원 정보 CSV `[0.7, 0.3]`(이름 등 고유명사가 많아 BM25 비중을 높임)**, 문서 양식 `[0.5, 0.5]`.

## 주의사항

- **임계값은 데이터마다 다시 튜닝해야 합니다.** 0.4는 이 실습 데이터 기준값입니다. 실제 문서로 정상/비정상 질문 각 10건을 돌려 경계를 찾으세요.
- **`EnsembleRetriever` import 경로가 버전에 따라 다릅니다.** langchain 1.x에서는 `langchain_classic.retrievers`에 있습니다. 위 코드의 try/except 패턴을 그대로 쓰세요.
- **BM25는 인메모리입니다.** 문서를 매번 다시 로드해 인덱스를 만들어야 하며, 대용량에서는 Elasticsearch/OpenSearch 같은 검색엔진으로 대체하는 것이 실무 방식입니다.
- **top-k를 무작정 늘리지 마세요.** 컨텍스트가 길어지면 비용이 오르고 LLM의 주의가 분산됩니다. 비용 관점 → [[개념_비용_라우팅_최적화]]
- 리랭커(Cross-Encoder) 도입은 이 교안 범위 밖입니다. 하이브리드로도 부족하면 다음 단계로 검토하세요.
