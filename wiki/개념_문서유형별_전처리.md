# 개념: 문서 유형별 RAG 전처리

## 한눈에 보기

RAG 성능은 모델보다 **적재 전 전처리**가 좌우합니다. 은행 내부자료(약관·FAQ·금리표·상담 스크립트)는 구조가 전부 다른데, 이를 무시하고 글자 수로만 자르면 조항이 중간에 끊기고 질문과 답변이 분리됩니다.
이 문서는 유형별로 다르게 자르는 4가지 기법과, 적재 전 PII 마스킹 절차를 정리합니다.

## 핵심 개념

| 문서 유형 | 의미 단위 | 일괄 분할하면 생기는 일 |
|---|---|---|
| **약관·내부규정** | 조항(제N조) | 조항이 중간에 잘려 "1개월 이내 연 0.1%" 같은 조건이 소실 |
| **FAQ** | Q&A 한 쌍 | 질문과 답이 다른 조각으로 분리되어 답만 검색됨 |
| **표(금리표·엑셀)** | 행 1개 | "12개월"과 "연 3.5%"의 대응 관계를 임베딩이 못 잡음 |
| **상담 스크립트** | 대화 턴 | 전화번호·계좌번호가 그대로 벡터DB에 적재됨 |

**핵심 원칙**: 조각(chunk) 하나가 **그 자체로 완결된 의미**를 가져야 합니다. 그리고 조각마다 `metadata`를 붙여야 나중에 필터 검색이 가능합니다.

## 전체 동작 흐름

```
원본 4종
 ├─ 약관        → 정규식 re.split(r"(?=제\d+조)")  → 조항 1개 = Document 1개
 ├─ FAQ         → 빈 줄("\n\n")로 분리            → Q&A 쌍 = Document 1개
 ├─ 금리표(CSV) → 행 → 자연어 문장으로 변환        → 문장 1개 = Document 1개
 └─ 상담스크립트 → PII 마스킹 후                   → Document 1개
        ↓
   전부 metadata(doc_type, 상품명, 조항, version) 부착
        ↓
   하나의 Chroma 컬렉션에 통합 적재
        ↓
   filter={"doc_type": "약관"} 처럼 유형별 검색 가능
```

## 핵심 코드

### 1. 나쁜 예 — 왜 문제인지 먼저 확인

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

all_text = terms_text + "\n" + faq_text + "\n" + rate_table_text
naive_chunks = RecursiveCharacterTextSplitter(chunk_size=150, chunk_overlap=20).split_text(all_text)
# → "제3조(중도해지 이율)" 조항이 두 조각으로 갈라져
#    "가입 후 1개월 이내 연 0.1%" 예외 조건만 따로 떨어져 나갑니다
```

"가입하고 3주 만에 해지하면 이율이 어떻게 되나요?"로 검색해 보면, 정답 예외 조건이 아닌 엉뚱한 조각이 잡힙니다.

### 2. 약관 → 조항 단위 분리 (재사용 가치 높음)

`(?=제\d+조)` 는 **전방탐색(lookahead)** 으로, "제N조" 앞에서 자르되 그 문자열은 남깁니다.

```python
import re
from langchain_core.documents import Document

clauses = [c.strip() for c in re.split(r"(?=제\d+조)", terms_text) if c.strip().startswith("제")]

term_docs = []
for clause in clauses:
    clause_no = re.match(r"제\d+조", clause).group()
    term_docs.append(Document(
        page_content=clause,
        metadata={
            "doc_type": "약관",
            "상품명": "KB 든든 정기예금",
            "조항": clause_no,          # 답변에서 "제3조에 따르면"이라고 인용 가능
            "version": "2026-01",       # 개정 이력 추적용
        },
    ))
```

> 💡 `version` 메타데이터는 실무에서 특히 중요합니다. 약관이 개정되면 구버전을 검색에서 제외해야 하는데, 이 필드가 없으면 통째로 재적재해야 합니다.

### 3. FAQ → Q&A 쌍 유지

```python
pairs = [p.strip() for p in faq_text.strip().split("\n\n") if p.strip()]
faq_docs = [
    Document(
        page_content=pair,                                    # Q와 A가 한 조각에
        metadata={"doc_type": "FAQ", "질문": pair.split("\n")[0].replace("Q: ", "")},
    )
    for pair in pairs
]
```

### 4. 표 → 행을 자연어 문장으로 (실무 표준 기법)

표를 통째로 넣으면 임베딩이 열-값 관계를 잡지 못합니다. **행 하나를 완성된 문장 하나로** 바꾸는 것이 정답입니다. 엑셀 자료도 같은 원리로 처리합니다.

```python
table_docs = []
for line in rate_table_text.strip().split("\n")[1:]:      # 헤더 제외
    product, period, rate = line.split(",")
    table_docs.append(Document(
        page_content=f"{product}의 {period} 기본금리는 {rate}입니다.",   # ← 자연어 문장
        metadata={"doc_type": "금리표", "상품명": product, "기간": period},
    ))
```

### 5. 적재 전 PII 마스킹

Vector DB에 들어간 민감정보는 **검색으로 누구에게나 노출**됩니다. 반드시 적재 전에 가립니다.

```python
import re

PII_PATTERNS = {
    "주민등록번호": r"\d{6}-\d{7}",
    "전화번호":     r"01[016-9]-?\d{3,4}-?\d{4}",
    "계좌번호":     r"\d{3}-\d{2,6}-\d{2,6}",
}

def mask_pii(text):
    found = []
    for name, pattern in PII_PATTERNS.items():
        if re.search(pattern, text):
            found.append(name)
            text = re.sub(pattern, f"[{name}-MASKED]", text)
    return text, found

masked_script, detected = mask_pii(script_text)
script_docs = [Document(
    page_content=masked_script,
    metadata={"doc_type": "상담스크립트", "pii_masked": True},   # 마스킹 여부를 기록
)]
```

### 6. 통합 적재 후 메타데이터 필터 검색

```python
good_store = Chroma.from_documents(term_docs + faq_docs + table_docs + script_docs,
                                   embeddings, collection_name="kb_preprocess_good")

# 약관 안에서만 검색
for doc in good_store.similarity_search("중도해지", k=2, filter={"doc_type": "약관"}):
    print(f"[{doc.metadata['조항']}]", doc.page_content[:60])
```

## 적재 전 품질 점검 체크리스트

- [ ] 조각 하나가 그 자체로 의미가 통하는가 (조항·Q&A가 끊기지 않았는가)
- [ ] 표는 행 단위 자연어 문장으로 변환했는가
- [ ] `doc_type`·`상품명`·`version` 메타데이터가 붙어 있는가
- [ ] PII를 적재 전에 마스킹했는가
- [ ] 대표 질문 3개로 검색했을 때 정답 조각이 상위에 오는가

## 주의사항

- **정규식은 문서 서식에 의존합니다.** `제N조` 표기가 `제 3 조`, `3조`처럼 흔들리면 정규식이 실패합니다. 실제 문서에 적용하기 전 몇 건을 눈으로 확인하세요.
- **FAQ 분리 기준(`\n\n`)도 마찬가지**입니다. 원본이 표 형태거나 번호 목록이면 다른 규칙이 필요합니다.
- **PII 정규식은 완전하지 않습니다.** 공백·점 구분(`010.1234.5678`), 여권번호, 이름+생년월일 조합 등은 잡히지 않습니다. 실무에서는 Microsoft Presidio 같은 전용 도구를 병행하세요 → [[개념_PII보호와_거버넌스]]
- **마스킹은 원본 보존과 함께 설계해야 합니다.** 마스킹된 텍스트만 남기면 나중에 원문 대조가 불가능합니다. 원본은 권한이 통제된 별도 저장소에 두는 것이 실무 방식입니다.
- 이 노트북의 샘플은 **전부 가상 데이터**입니다. 실제 고객 데이터로 실습하지 마세요.
