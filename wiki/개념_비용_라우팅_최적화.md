# 개념: 비용·라우팅 최적화 (Agent Ops)

## 한눈에 보기

"영업시간이 언제예요?" 같은 간단한 질문까지 가장 비싼 모델로 처리하면, 모든 우편물을 특송으로 보내는 것과 같습니다.
저비용 모델이 **난이도를 먼저 분류**하고 알맞은 모델로 보내는 **라우팅**, 그리고 같은 질문에 다시 계산하지 않는 **캐싱** — 이 둘로 운영비를 크게 줄일 수 있습니다.

## 핵심 개념

| 기법 | 원리 | 효과 |
|---|---|---|
| **난이도 라우팅** | 싼 모델이 simple/complex/sensitive 분류 → 알맞은 모델로 | 단순 질문 비용을 1/10 수준으로 |
| **캐싱** | 같은 질문의 답을 저장해 재사용 | 반복 질문 비용 0원 |
| **비용 장부(CostTracker)** | 호출마다 모델·토큰·금액 기록 | 임계값 알림과 연결 |

**왜 중요한가**: Agent는 답변 하나에 LLM을 여러 번 호출합니다(분류 → 검색 → 생성 → 검증). 하루 1만 건 상담이면 호출은 수만 번 — **모델 선택이 곧 운영비**입니다.

**난이도 3분류**

| 분류 | 뜻 | 라우팅 |
|---|---|---|
| `simple` | 단순 FAQ (영업시간, 재발급 절차) | 저가 모델 + **캐시 대상** |
| `complex` | 규정·계산이 필요한 질문 | 고성능 모델 |
| `sensitive` | 투자 권유·민원 등 민감 질문 | 고성능 모델 + **출력 필터 강제** |

## 전체 동작 흐름

```
[질문]
   ↓ 캐시 조회 ── 적중 → 비용 0원, 즉시 반환
   ↓ 미스
[저가 모델 라우터] → RouteDecision(difficulty, reason)   ← 분류 비용은 아주 작음
   ↓
   ├─ simple    → 저가 모델 → 답변 → 캐시에 저장
   ├─ complex   → 고성능 모델 → 답변
   └─ sensitive → 고성능 모델 → 답변 → 금지어 필터 → 걸리면 [차단 + 상담원 연결]
   ↓
[CostTracker] 질문·모델·토큰·금액 기록
```

## 핵심 코드

### 1. 모델 2단 구성과 단가표

```python
CHEAP_MODEL  = "gpt-4o-mini"
STRONG_MODEL = "gpt-4.1"

cheap  = ChatOpenAI(model=CHEAP_MODEL, temperature=0)
strong = ChatOpenAI(model=STRONG_MODEL, temperature=0)

# 1000토큰당 원화 (교안 실습 기준값 — 실제 단가는 반드시 최신 가격표로 갱신)
PRICE_TABLE = {
    CHEAP_MODEL:  {"input": 0.2, "output": 0.8},
    STRONG_MODEL: {"input": 2.8, "output": 11.2},
}
```

### 2. 호출 1건의 비용 계산 ⭐

```python
def cost_of(response, model_name):
    usage = response.usage_metadata or {}
    input_tokens  = usage.get("input_tokens", 0)
    output_tokens = usage.get("output_tokens", 0)
    price = PRICE_TABLE[model_name]
    won = input_tokens / 1000 * price["input"] + output_tokens / 1000 * price["output"]
    return won, input_tokens, output_tokens
```

> `usage_metadata` 는 LangChain이 표준화한 필드입니다. 모델·API에 따라 비어 있을 수 있으니 기본값 처리를 두세요.

### 3. CostTracker — 비용 장부 ⭐

```python
class CostTracker:
    def __init__(self, name):
        self.name, self.records = name, []

    def add(self, question, model_name, won, input_tokens, output_tokens, cached=False):
        self.records.append({"question": question, "model": model_name, "won": won,
                             "input_tokens": input_tokens, "output_tokens": output_tokens,
                             "cached": cached})

    def total_cost(self):
        return sum(r["won"] for r in self.records)

    def report(self):
        print(f"===== 비용 리포트: {self.name} =====")
        for r in self.records:
            tag = " (캐시 재사용)" if r["cached"] else ""
            print(f"  {r['won']:7.2f}원 | {r['model']:<22} | {r['question'][:30]}{tag}")
        print(f"  합계: {self.total_cost():.2f}원 (호출 {len(self.records)}건)\n")
```

`AgentMonitor`(오류율·지연)와 같은 원리이며, 시간당 비용 임계값 알림과 바로 연결됩니다 → [[개념_PII보호와_거버넌스]]

### 4. 난이도 분류 라우터 ⭐⭐

`include_raw=True` 를 쓰면 파싱 결과와 원본 응답을 함께 받아 **라우터 자체의 비용도 측정**할 수 있습니다.

```python
from typing import Literal
from pydantic import BaseModel, Field

class RouteDecision(BaseModel):
    reason: str = Field(description="이 난이도로 분류한 이유 (30자 이내)")
    difficulty: Literal["simple", "complex", "sensitive"] = Field(
        description="simple=단순 FAQ, complex=규정·계산이 필요한 질문, sensitive=투자권유·민원 등 민감 질문")

router_llm = cheap.with_structured_output(RouteDecision, include_raw=True)   # ← 비용 측정용

def route_and_answer(question, tracker):
    routed = router_llm.invoke(ROUTE_PROMPT.format(question=question))
    decision = routed["parsed"]
    won, i, o = cost_of(routed["raw"], CHEAP_MODEL)
    tracker.add(question, CHEAP_MODEL + " (라우터)", won, i, o)

    model_name, worker = (CHEAP_MODEL, cheap) if decision.difficulty == "simple" else (STRONG_MODEL, strong)
    response = worker.invoke(ANSWER_PROMPT.format(question=question))
    answer = response.content
    won, i, o = cost_of(response, model_name)
    tracker.add(question, model_name, won, i, o)

    if decision.difficulty == "sensitive":                    # 민감 질문은 출력 필터 강제
        for word in FORBIDDEN:
            if word in answer:
                answer = "[차단] 금융 규정상 사용할 수 없는 표현이 포함되어 답변을 보류합니다. 상담원을 연결해 드리겠습니다."
                break
    return answer, decision.difficulty
```

### 5. 캐싱 — simple 질문만 ⭐

```python
answer_cache = {}

def cached_route_and_answer(question, tracker):
    key = question.strip()
    if key in answer_cache:
        tracker.add(question, "cache", 0.0, 0, 0, cached=True)
        return answer_cache[key]
    answer, difficulty = route_and_answer(question, tracker)
    if difficulty == "simple":              # ← simple만 캐시
        answer_cache[key] = answer
    return answer
```

> ⚠️ **고객별로 답이 달라지는 질문(잔액·한도·거래내역)은 절대 캐시하면 안 됩니다.** 다른 고객의 답이 나가는 사고가 됩니다. 교안이 명시적으로 경고하는 지점이며, "누구에게나 같은 답인 FAQ만" 캐시한다는 전제입니다.

### 6. A/B 비교

```python
tracker_a = CostTracker("A. 전부 고성능 모델")
for q in QUESTIONS:
    response = strong.invoke(ANSWER_PROMPT.format(question=q))
    tracker_a.add(q, STRONG_MODEL, *cost_of(response, STRONG_MODEL))

tracker_b = CostTracker("B. 라우팅 + 캐시")
for q in QUESTIONS:
    cached_route_and_answer(q, tracker_b)

cost_a, cost_b = tracker_a.total_cost(), tracker_b.total_cost()
print(f"결론: {cost_a:.2f}원 → {cost_b:.2f}원, 약 {(1 - cost_b/cost_a)*100:.0f}% 절감")
```

## Agent Ops 체크리스트

- [ ] 단가표가 **최신 가격**으로 갱신되어 있는가
- [ ] 라우팅에 저가 모델을 쓰고 있는가 (라우팅을 비싼 모델로 하면 의미가 없음)
- [ ] 캐시 대상이 **고객 무관 FAQ**로 한정되어 있는가
- [ ] 캐시에 만료(TTL)가 있는가 — 금리·정책이 바뀌면 낡은 답이 나감
- [ ] 시간당 비용 임계값 알림이 있는가
- [ ] 루프 상한이 있는가 — 무한 루프는 곧 무한 비용 → [[개념_Agent_통제력과_루프탈출]]
- [ ] top-k와 컨텍스트 길이가 과하지 않은가 → [[개념_검색품질_개선]]

## 주의사항

- 🚨 **`Day05_07` 노트북에 실제 OpenAI 키가 하드코딩되어 있습니다** → [[오류_환경설정_트러블슈팅]]
- ⚠️ **`PRICE_TABLE` 값은 실습용 예시**입니다. 실제 단가는 모델·시점·계약에 따라 다르므로 **반드시 최신 가격표로 갱신**하세요. 노트북 수치를 그대로 인용해 예산을 산정하지 마세요.
- ⚠️ **노트북의 절감률 출력값은 실행 시점 응답 길이에 따라 달라집니다.** 특정 퍼센트를 사실로 인용하지 마세요.
- **캐시 만료가 없습니다.** 예제의 `answer_cache` 는 무한 보관 dict입니다. 금리·수수료가 바뀌면 틀린 답을 계속 내보냅니다. TTL과 무효화 정책을 반드시 추가하세요.
- **캐시 키가 문자열 완전 일치**입니다. "영업시간이 언제예요?"와 "영업시간 알려주세요"는 다른 키가 되어 적중률이 낮습니다. 실무에서는 정규화나 임베딩 유사도 기반 시맨틱 캐시를 씁니다.
- **라우터가 오분류하면 품질이 떨어집니다.** complex 질문을 simple로 잘못 분류하면 싼 모델이 부실한 답을 냅니다. 오분류율을 측정하세요 → [[개념_Agent_평가]]
- **비용 절감보다 사고 방지가 우선입니다.** sensitive 분류에서 출력 필터를 붙이는 이유가 이것입니다 → [[개념_PII보호와_거버넌스]]
- **프롬프트 캐싱·배치 API 등 벤더 제공 기능**은 이 교안 범위 밖입니다. 반복되는 긴 시스템 프롬프트가 있다면 추가 검토 대상입니다.
