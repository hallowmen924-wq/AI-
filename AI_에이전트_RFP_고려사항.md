## 목표
- S3에 보관된 시행문을 일별로 수집하여 부점별로 요약·배포한다.
- 회사 EAM(권한관리시스템)을 권한 원천(PDP)으로 활용해 부점별 조회권한을 실시간 결정 및 강제한다.
- 검색(벡터 DB) 수준에서의 접근 차단을 보장하고 모든 조회·요약·전송 활동을 감사 가능하게 기록한다.

## 핵심 요구 요약
- **권한 결정**: 중앙 EAM(PDP)으로 실시간 권한 질의(REST). 응답: allow|deny, ttl, reason.
- **엔포스먼트(필수)**: `RetrievalStore`(벡터 DB) adapter 내부에서 권한검증 수행(쿼리 실행 전 차단).
- **보조**: S3 정책으로 원문 다운로드 제어(원본 보호), 에이전트 레벨에서는 방어적 필터 주입 및 사용자 메시지 처리.
- **감사**: PDP 질의·응답, 검색 쿼리(필터), 반환 문서 IDs, 사용자, 타임스탬프를 SIEM으로 전송.

## 1. 시스템 아키텍처 및 흐름

1) **인게스트**: 원문(.docx/.md/.txt) + 메타를 S3에 업로드. 메타에는 `branch`, `receiver`, `sender`, `date` 포함.
2) **파싱·청킹**: `ParsedDocument` → `TextChunk` → `EmbeddedChunk`(임베딩 수행).
3) **업서트**: `EmbeddedChunk`를 `RetrievalStore.upsert(...)`로 저장(LanceDB / Postgres+pgvector 등).
4) **요청 시 권한검증**: Agent/API가 토큰→사용자 식별 후 EAM에 `subject/action=read/resource{branch}` 질의.
5) **저장소 엔포스먼트(Primary)**: `RetrievalStore.search(...)` 내부에서 EAM 결정을 확인하고, 허가되지 않으면 빈 결과 또는 예외 반환.
6) **검색·요약**: `Retriever`가 허용 필터로 검색 → LLM 요약 → 전송. 모든 단계 로깅.

### S3 vs RetrievalStore 역할 구분
- **S3**: 원문(오리지널 파일) 저장, 버킷 정책/IAM으로 원문 접근 제어에 적합.
- **RetrievalStore**: 검색·리트리벌(청크/벡터) 저장소 — 실제 검색 응답을 결정하므로 권한 검증의 결정적 지점.

### 옵션 비교
| 옵션 | 설명 | 장점 | 단점/리스크 |
|---|---|---|---|
| (A) S3 레벨 권한통제만 | 버킷/IAM 정책만 사용 | 구현 쉬움 | 검색 결과(벡터 DB) 유출 차단 불가 → 보조 수단으로만 권고 |
| (B) Agent-embedded EAM 호출 | 에이전트 코드 내에서 직접 PDP 호출 | PoC에 적합 | 권한 로직 분산·우회 위험 → 단기 PoC용 |
| (C) 중앙 PDP(EAM) + RetrievalStore 수준 엔포스먼트 **(권장)** | 저장소 어댑터 내부에서 강제 | 보안·감사·운영 일관성 우수 | 추가 인프라·캐시 필요 |

**권장 결정**: 운영환경은 중앙 PDP(EAM) 연동 + RetrievalStore 내부 엔포스먼트 + Agent 방어층 + S3 원문 보호 **병행**.

## 2. 기능·비기능 요구사항 (NFR)

- **PDP API**: `POST /pdp/authorize {subject, action:'read', resource:{type:'document', branch, id}}` → `{decision, ttl, reason}`
- **캐시**: TTL 기반 권한 캐시(예: Redis)로 성능 보완
- **장애정책**: 기본 Fail-Closed(안전) — 운영에서 Fail-Open 옵션 제공
- **보안**: TLS1.2+, mTLS 옵션, 토큰 마스킹, 민감정보 마스킹 규칙
- **성능 목표**: PDP 평균 응답 <50ms(목표), 최대 허용 200ms(캐시 포함)

## 11. 확인/질문 사항 (담당자별 · IT관점 / 현업관점)

### 11-1. 김경덕 프로 — 공통모듈 담당

**IT관점**
- [ ] `RetrievalStore.search()` 내부 권한 훅은 기존 pgvector/LanceDB provider 인터페이스에 어떤 형태로 삽입되나? (미들웨어 vs 코드 수정)
- [ ] 권한 필터(`branch` 등)를 벡터 검색 쿼리(예: pgvector의 WHERE절/metadata filter)에 어떻게 반영하나? 성능 저하 없이 가능한가?
- [ ] Fail-Closed 기본 정책을 공통모듈 레벨에서 어떻게 구현/테스트하나? (PDP 응답 지연·장애 시나리오 포함)
- [ ] 캐시(Redis TTL) 계층은 공통모듈에 내장되나, 별도 인프라 컴포넌트인가?
- [ ] 기존 1차 제공 범위(LLM Client/Embedding/Retrieval)와 이번 EAM 훅 작업의 버전/릴리즈 관계는?
- [ ] **시행문 저장용 임베딩 모델/청킹·리트리버 기법을 자유롭게 선택해도 되는지, 아니면 기존 프로모션 에이전트 프로젝트에서 쓰던 모델/기법을 그대로 재사용해야 하는지?** (재사용이라면 모델명, 청킹 전략, hybrid search 설정까지 동일하게 맞춰야 하는지 확인 필요)
- [ ] 재사용해야 한다면, 프로모션 에이전트와 시행문의 문서 특성(길이, 구조, 표/서식 포함 여부 등) 차이로 인해 청킹 전략(RecursiveTokenChunker 파라미터 등)을 별도로 튜닝해야 하는지?

**현업관점**
- [ ] 부점 담당자가 "왜 이 문서가 검색이 안 되는지" 문의했을 때, 공통모듈 로그만으로 원인 추적이 가능한가?
- [ ] 권한 변경(인사이동 등) 시 반영 시점은 실시간인가, 배치 주기인가?

### 11-2. 김강열 프로 — 전반 총괄 엔지니어

**IT관점**
- [ ] EAM(PDP) API 스펙 문서(요청/응답 필드, 인증 방식)는 확보되어 있나?
- [ ] PDP 장애 시 Fail-Open 전환 절차와 승인권자는 누구인가?
- [ ] PoC-A → PoC-B → 운영화 단계별 Go/No-Go 판단 기준은?
- [ ] SIEM 연동 대상 시스템(제품/버전)과 전송 프로토콜(syslog/API 등)은 확정되었나?
- [ ] 성능 목표(PDP 평균 <50ms, 최대 200ms)는 캐시 적중률을 전제로 한 수치인가? 캐시 미스 시나리오 목표치는?

**현업관점**
- [ ] 전체 일정(제안 접수 2주 + PoC 4~6주 + 개발 8~12주 + 시범배포 2주) 중 현업(부점) 검증 참여 시점은 언제인가?
- [ ] 시범배포 대상 부점 선정 기준과 확대 로드맵은?
- [ ] 장애/오탐(잘못된 접근 차단 또는 허용) 발생 시 에스컬레이션 창구는?

### 11-3. 동명환 프로 — S3 담당자

- [ ] 인게스트 메타데이터(`branch`, `receiver`, `sender`, `date`) 스키마는 누가 확정/검증하나? 필수값 누락 시 처리 정책은?

- [ ] 원문 자체 열람 권한(S3)과 요약본 열람 권한(RetrievalStore)이 다르게 설정될 수 있는 케이스가 있는가?
